## Findings

### 1. High - AutoSim snapshots can leak filled auth secrets into LLM page-state payloads

Evidence:

- `prototype/lib/trails-browser-page.ts:110` builds the compact snapshot element name.
- `prototype/lib/trails-browser-page.ts:111-115` falls back to `(el as HTMLInputElement).value`.
- `prototype/lib/trails-author.ts:329-330` sends `domSnapshot: dom` to the author model.
- `prototype/lib/trails-author.ts:472-475` resolves credential placeholders and fills the real value into the page.

Why this matters:

ADR-0001 is respected in the trajectory/log path, but the next snapshot can still expose a live OTP/password if the filled input has no `aria-label`, `placeholder`, text, `name`, or `title`. For password/OTP forms this is common enough to be a real leak: after AutoSim fills `{{cred:autosim-auth:otp}}`, the next `krefSnapshot()` can include the resolved secret as the textbox name and send it to the LLM.

Concrete fix:

- Never include `.value` for `input`, `textarea`, or `select` in `krefSnapshotBody`.
- For inputs, derive names only from label/ARIA/placeholder/name/title, and emit a generic `textbox ""` when no safe name exists.
- Explicitly mask `type=password`, OTP-ish fields, and any field filled from a credential placeholder.
- Apply the same rule to `prototype/lib/trails-snapshot.ts:103-108`, which has the same fallback.
- Add a unit test that fills an unlabeled password/OTP input with a known secret and asserts the snapshot does not contain it.

### 2. High - `mint_link` auth accepts arbitrary navigation URLs and does not enforce signed-token expiry/replay

Evidence:

- `prototype/lib/autosim-auth-exec.ts:88-99` allows the stored secret to be a full absolute URL, a path, or a bare token.
- `prototype/lib/autosim-auth-exec.ts:121-126` navigates to the resulting URL and returns `established: true` without checking the URL shape or session result.
- `prototype/lib/trails-author.ts:259-260` runs that navigation at authoring start.

Why this matters:

The comments describe a signed `/test-login?token=...` session-mint path, but this implementation accepts any full URL. A project auth setup token holder can make the server-side browser visit arbitrary origins, including internal or metadata endpoints reachable from the 1 GB prod box. There is also no enforcement here that the token is env-gated, short-lived, single-use, or replay-protected; the safety is entirely delegated to whatever URL was stored.

Concrete fix:

- Store `mint_link` as an opaque token or same-origin path only, not an arbitrary absolute URL.
- Resolve only against `baseUrl.origin`, require pathname `/test-login`, and reject private/link-local/loopback origins unless an explicit test env flag is set.
- Define the token format with `exp`, `jti`, audience/project, and HMAC or public-key signature; consume `jti` once server-side to prevent replay.
- After navigation, verify that the session is actually established before returning `established: true`.
- Add tests for absolute URL rejection, expired token rejection, replay rejection, and successful same-origin mint.

### 3. Medium - `needs_auth` resume has no zombie/double-resume guard despite the code comment calling it out

Evidence:

- `prototype/lib/trails-author.ts:697-703` says a future guard should expire old `needs_auth` rows and prevent double-resume.
- `prototype/lib/trails-author.ts:765-768` loads any prior session checkpoint by id, with no status, age, or already-resumed check.
- `prototype/lib/trails-author.ts:783` creates a new resumed session unconditionally once a checkpoint exists.

Why this matters:

The UI listing hides old paused sessions after seven days, but the backend resume path can still be called directly with an old or already-resumed session id. That can spawn duplicate authoring drives from the same checkpoint or resume stale state long after the page/app/auth context has changed.

Concrete fix:

- In `runAuthorNow`, require the prior session to be `stalled` or `needs_auth`, updated within the TTL, and to have a checkpoint.
- Atomically mark the prior session as `resuming` or record a `resumed_by` before starting the new drive.
- Reject if another session already has `resumed_from = priorId` in `running`, `stalled`, `needs_auth`, or `crystallized`.
- Add tests for stale resume rejection and double-resume rejection.

### 4. High - Any project member can invite external users by assigning a ticket, bypassing the admin-only invite path

Evidence:

- `prototype/server.ts:4377` says any project member may edit assignee.
- `prototype/server.ts:4390-4394` accepts any valid assignee email.
- `prototype/server.ts:4426-4435` calls `notifyTicketAssignee` for a new assignee.
- `prototype/server.ts:89-101` creates a pending assignment invite for non-members and emails a login link.
- `prototype/lib/db.ts:1652-1668` accepts all pending assignment invites for that email and calls `addProjectMember`.
- `prototype/server.ts:5143-5149` keeps the normal project invite endpoint admin-only, so assignment is a privilege bypass.

Why this matters:

A plain project member can add any external email to the project by assigning a ticket to it. When that email signs in, `acceptPendingTicketAssignmentInvites` auto-attaches them as a project member without an admin action or signed invite token. This bypasses the existing admin-only `/api/projects/:id/invite` control.

Concrete fix:

- Require project admin access when assigning to an email that is not already a project member.
- Alternatively, create a signed, single-use invite token that an admin must authorize before auto-attach.
- Keep member-to-member assignment allowed, but block member-to-external assignment.
- Add tests: member assigning existing member succeeds; member assigning external email returns 403; admin assigning external email creates the pending invite.

### 5. Medium - Deleting a label can remove cross-project ticket-label associations before ownership is checked

Evidence:

- `prototype/server.ts:5121-5124` calls `deleteLabel(proj.id, lid)` and returns success without checking the result.
- `prototype/lib/db.ts:3538-3544` first deletes all `ticket_labels` rows for `label_id`, then deletes the label with `WHERE id=? AND project_id=?`.

Why this matters:

If an admin of project A submits a label id from project B, the label row is not deleted because of the project guard, but the associations for that label are already removed globally. That is cross-project data corruption.

Concrete fix:

- In `deleteLabel`, first verify `SELECT 1 FROM labels WHERE id=? AND project_id=?`.
- Only delete `ticket_labels` after ownership is confirmed.
- Return false for non-owned labels and have the route return 404.
- Prefer one transaction or a scoped delete such as `DELETE FROM ticket_labels WHERE label_id IN (SELECT id FROM labels WHERE id=? AND project_id=?)`.
- Add a regression test with two projects and a foreign label id.

### 6. Medium - Bulk ticket actions are partially applied but reported as a single success

Evidence:

- `prototype/server.ts:5031-5085` loops ticket ids one by one.
- `prototype/server.ts:5040` updates metadata per ticket.
- `prototype/server.ts:5081-5082` catches and suppresses label attach/detach failures.
- `prototype/server.ts:5085` returns `{ ok: true, updated }` even if labels failed on some rows.

Why this matters:

The ticket asked for "atomically-ish" bulk actions. The current implementation can leave a mixed state: metadata may update, label operations may silently fail, and the client still receives success. That is a correctness risk for bulk close/priority/label workflows and makes audit/debugging hard.

Concrete fix:

- Validate all target ticket ids and label ids up front.
- Apply the operation in a transaction where the DB supports it, or return per-ticket/per-operation results with explicit failures.
- Do not swallow label attach/detach failures; include them in the response and avoid claiming full success.
- Add a test where one label operation fails and the endpoint does not return silent success.

### 7. Medium - Past-due Stripe subscriptions are treated as paid plans before enforcement is enabled

Evidence:

- `prototype/server.ts:1045-1048` returns the requested plan for `active`, `trialing`, and `past_due`.
- `prototype/lib/billing.ts:23-29` correctly keeps quota metadata dark unless `KLAV_BILLING_ENFORCEMENT === "1"`.

Why this matters:

There is no accidental live enforcement today, which is good. But once enforcement is enabled, `past_due` subscriptions will retain Pro/Team quotas indefinitely until Stripe sends a terminal cancellation/deletion update. That weakens the subscription lifecycle gate.

Concrete fix:

- Before enabling `KLAV_BILLING_ENFORCEMENT`, decide an explicit grace policy.
- If no grace is intended, map `past_due` to `free`.
- If grace is intended, persist `past_due_since` and enforce a bounded grace window.
- Add webhook tests for `active`, `trialing`, `past_due`, `canceled`, and `unpaid`.
